# AgenDao 三前端统一事件契约（阶段记录）

这份文档记录的是前端事件语义收口过程中的一个阶段切面。它解释当时为什么要统一 transcript authority、progress state 和 reconcile fallback，但不单独定义今天的产品真相。

当前需要了解的正式产品面，请优先阅读：

- `docs/index.md`
- `docs/commands.md`
- `docs/documentation-status.md`

下面保留的是当时的契约目标与收口分析：

1. 哪些事件是 transcript 权威输入
2. 哪些事件只是 progress / modal 状态
3. `session.updated` 何时只能作为 reconcile fallback
4. 当前三端偏差在哪里
5. 后续收口顺序是什么

---

## 1. Canonical Event Contract

唯一正式事件契约：`ServerEvent / StateChange`

共享事实：

1. `output_block` 是实时内容主通道
2. `question.created / question.resolved` 是交互状态通道
3. `permission.requested / permission.resolved` 是权限状态通道
4. `tool_call.started / tool_call.completed` 是工具生命周期通道
5. `topology.changed` 是拓扑/阶段图通道
6. `session.updated` 是 reconcile fallback，不是主渲染通道

---

## 2. 统一语义

### 2.1 Transcript-bearing

只有以下内容允许进入会话 transcript：

1. `output_block.message` 且 `live_identity.part_kind = assistant_text`
2. `output_block.reasoning` 且 `live_identity.part_kind = assistant_reasoning`
3. `output_block.tool` 的最终结果，或 `live_identity.part_kind = tool_result`
4. 无 `live_identity` 的兼容路径下，只接受 `full / done / error / result` 这类完整快照

统一规则：

1. 同一 `{message_id, part_key}` 只能原位更新，不能重复产生 header
2. `End`/最终快照负责关闭 open slot，并让 block 立即可见
3. `tool start/running` 不属于 transcript
4. `scheduler_stage` 不属于 transcript

### 2.2 Progress-state Only

以下事件只更新运行态、活动态、topology 或 modal，不得写入 transcript：

1. `tool_call.started`
2. `tool_call.completed`
3. `output_block.scheduler_stage`
4. `session.busy`
5. `session.retrying`
6. `question.created / resolved`
7. `permission.requested / resolved`
8. `usage`
9. `topology.changed`

统一规则：

1. permission/question modal 只改变交互状态，不阻塞 `output_block` 消费
2. scheduler stage 只改变阶段卡、attached-session、topology，不生成 assistant message
3. tool running 只改变 activity/pending 状态，不生成最终 transcript block

### 2.3 Reconcile Fallback

`session.updated` 只允许在非高频 source 上触发全量重建。

必须跳过：

1. `prompt.stream`
2. `stream.prompt`
3. `prompt.scheduler.stage.content`
4. `prompt.scheduler.stage.reasoning`
5. `prompt.scheduler.stage.child.final`
6. `topology`

统一规则：

1. reconcile 只能补齐 persisted history 与 live state 的差值
2. reconcile 不得回放已显示 `USER`
3. reconcile 不得覆盖已经更“新”的 live transcript

---

## 3. 三端权威落点

### 3.1 TUI

当前最接近目标权威：

1. `output_block` 进入 `SessionContext.apply_output_block_incremental()`
2. 当前 session view 直接读取 `SessionContext`
3. `permission/question` 只进入独立 prompt/runtime 队列
4. `session.updated` 仅走 `handle_session_updated_reconcile()`

结论：

1. TUI 是当前三端的基准实现
2. 后续契约以 TUI 语义为准，不再由 CLI 或 Web 反向定义

### 3.2 Web

当前主链：

1. `useServerEventStream.ts` 消费 server events
2. `useTranscriptFeedState.ts` 合并 `history + live output_block`
3. `question / permission` 进入单独 interaction state
4. `scheduler_stage` 与 tool activity 进入 execution activity / runtime surface

当前优点：

1. 已明确区分 transcript-bearing 与 non-transcript live
2. 已有 “history 覆盖 live block” 的治理逻辑
3. 已有 tool/scheduler 的独立活动面

当前风险：

1. `history + live feed + activity` 仍然是多块读面，必须持续防止重复 authority
2. compatibility 路径仍需继续收窄，避免 generic fallback 把非 transcript block 混进正文

### 3.3 旧 CLI rich 前端

这部分描述的是旧 CLI rich 交互前端仍存在时的偏差来源；它用于解释为什么后来需要继续把交互责任下沉到更稳定的读面。

1. 当时的 rich 前端存在多层 authority：`rich state`、`root/attached transcript`、`projection.transcript`
2. prompt 会话栏读取 `projection.transcript`
3. terminal scrollback 读取 `render_rich_state()`
4. modal 交互一度会阻塞 SSE 主循环，说明交互状态与事件消费曾耦合过深

结论：

1. 当时最需要收口的是旧 CLI rich 前端的多权威问题
2. 这也是后续不再继续扩写那套局部兼容状态机的原因

---

## 4. 当前偏差清单

### 旧 CLI rich 前端

1. session 栏与 terminal scrollback 不是同一 authority
2. rich state 与 projection transcript 之间可能出现同步时序差
3. compatibility 路径比 TUI 更复杂

### TUI

1. 作为基准权威，重点不是重写，而是保证别的前端向它收敛

### Web

1. live feed 与 history rebuild 的边界要继续保持单向治理
2. activity panel / transcript panel / interaction panel 不得各自复制事件判定

---

## 5. 收口原则

统一原则只有一句：

> 每个前端都只能有一个“当前可见 transcript authority”；其他层只能缓存或适配，不能二次决定事件语义。

具体要求：

1. 各前端都不得自创 transcript-bearing 事件类型
2. 各前端都不得把 permission/question modal 变成事件泵阻塞点
3. 各前端都不得把 scheduler stage 写进最终 transcript
4. 各前端都不得依赖 `session.updated` 才让 block 可见

---

## 6. 逐前端收口顺序

### 第 1 项：旧 CLI rich 前端

目标：

1. 把 prompt 会话栏与 scrollback 收敛到同一 transcript authority
2. 删除旧 CLI rich 前端残留的多权威写入点
3. 让 permission/question modal 完全脱离 `output_block` 实时消费

完成标准：

1. permission 弹窗期间，`thinking / tool / assist` 仍持续出现在 session 栏
2. 整轮结束后不再“统一补吐”
3. `session.updated` 不再承担主渲染职责

### 第 2 项：Web

目标：

1. 收窄 compatibility route
2. 确认 `history + live feed` 只有一处 merge authority
3. activity/interaction 面板只读衍生状态，不复制 transcript 判定

完成标准：

1. live transcript 与 persisted history 不互相回滚
2. tool running / scheduler stage 不进入 transcript feed
3. question/permission 面板不影响 live transcript 增量显示

### 第 3 项：TUI

目标：

1. 保持基准实现地位
2. 只做对照审计，不重写主链

完成标准：

1. TUI 继续作为统一契约 reference
2. CLI/Web 的行为与 TUI 对齐，而不是 TUI 反向迁就其他端

---

## 7. 验收口径

三端统一验收只看三件事：

1. `thinking / final tool result / assist` 是否在块完成时立即出现
2. `permission/question` 是否只影响 modal 状态，而不阻塞 transcript 增量
3. `session.updated` 去掉后，主体验是否仍成立；若不成立，说明还有隐藏的第二权威
