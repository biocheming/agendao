## 总体评价

AgenDao 项目是 CLAUDE.md 道纪原则的高度忠实实现。五层架构 (Adapters → Orchestration → Session → Domain Services → Infrastructure) 与木火土金水五律精确对位。大部分核心治理原则已经落地为具体的类型、模块边界和运行时约束。

**核心病灶只有两条**（按优先级）：
1. **Prompt authority 还不够收口** — 分层构造存在但共享块复用做得不错，问题在于统一权威不够强
2. **Skill 语义投影关系还不够显式** — SchedulerSkillRef 没有声明自己是 SkillMeta 的投影

## 第一条：阴阳总律 — 合规度 90%

每个活跃语义域都有明确的阴阳对位。未发现"孤阳"或"孤阴"反模式。

| 语义域 | 阳面（生发/执行） | 阴面（收束/记账） |
|--------|-------------------|-------------------|
| 输入 | `normalize_ingress_source()` (`session/src/prompt/ingress.rs:30`) | `IngressStabilizationMetadata` |
| 执行 | `OrchestrationCore::execute_prompt()` (`orchestrator/src/core.rs:79`) | `LoopPolicy` + `EventBusTelemetry` |
| 工具调度 | `ToolRunner::execute_tool_call()` (`orchestrator/src/tool_runner.rs:46`) | `ToolMemoryObservation` → `MemoryAuthority` |
| 输出 | `StageSummary` + `StageEvent` (`stage-protocol/src/lib.rs:31`) | `OutputBlock` 统一类型 + `EventBusTelemetry` |
| 配置 | `ConfigStore::patch()` (单写点) | `ConfigStore::config()` (ArcSwap 无锁读) |

## 第二条：木律 (输入之木，贵在生发，忌多头生长) — 合规度 90%

### 合规项 ✅

- `normalize_ingress_source()` 将 cli/tui/web/api/scheduler 五源归一化为统一 `IngressSource` 枚举
- `IngressTurnEnvelope` 封装所有输入路径，包含稳定性元数据
- 输入变体 (slash command / attachment / replay) 复用同一 ingress 权威

### 注意项（非问题）

CLI 侧 `build_prompt_message()` (`cli/src/run/session_exec/message_io.rs:1`) 在本地做 command/input 拼接，然后将结果通过 `PromptRequest.message` 送入 session。`PromptRequest` 同时也保留了 `command` 字段。这不是"半草稿状态"问题（message_io 只是格式化，不持有状态），但如果未来 command/input 语义需要更精细的分离处理，将拼接逻辑完全移到 ingress 层会更干净。

## 第三条：火律 (执行之火，贵在点燃，忌多炉并起) — 合规度 95%

### 合规项 ✅

- **唯一执行内核**: `OrchestrationCore` 注释明确标注 "唯一执行内核"
- **单一 LLM 循环**: `runtime::run_loop` 是所有 preset 的共享循环实现
- **统一权限裁决**: `PermissionEngine` 处理所有 permission class/lifetime/matcher
- **统一工具调度**: `ToolRunner` 是唯一工具执行入口，包含 `repair_tool_call_name` 归一化
- **Home/Session 路径同构**: 通过 `local_server_bridge::local_prompt()` 统一入口

### 注意项（非问题）

Provider 层 `options()` (`provider/src/transform/options.rs:9-66`) 对不同 provider (openai, openrouter, baseten, zai/zhipuai) 的请求选项使用过程式 if-else 构造。注意：**这是 provider request options，不是 prompt text surface**。它属于执行层适配分支，未来可考虑迁移到 provider profile 的声明式配置中，降低分支复杂度，但不构成当前的道纪违规。

## 第四条：土律 (承载之土，贵在归一，忌地脉分裂) — 合规度 80%

这是发现最集中的领域。以下诊断经过 biocheming 逐代码点审核修订。

### 合规项 ✅

- **唯一配置真相**: `ConfigStore` 用 `ArcSwap` 实现无锁读取 + 单一 `patch()` 写点 (`config/src/store.rs`)
- **唯一状态所有权**: 每个状态域有独立 authority (MemoryAuthority, SkillAuthority, UserStateAuthority)
- **副作用经由编排层**: CLI 通过 `local_server_bridge` 走 server → orchestrator 路径

### 问题 #1：Prompt Surface 分层构造，统一权威还不够强 🟡

**现状**：prompt surface 由以下层次分别构造：

| 层次 | 位置 | 职责 |
|------|------|------|
| System prompt | `session/src/system.rs:20` | 两层系统提示 (产品 header + compatibility overlay) |
| Session prompt | `session/src/prompt/mod.rs` | 消息序列化、compaction、ingress |
| Preset dynamic prompt | `orchestrator/src/scheduler/presets/{sisyphus,atlas,hephaestus,prometheus}/prompt.rs` | 各 preset 的行为指令 |

**正面**：各 preset 的 prompt 构造**已经明显复用共享构件**：
- `orchestrator/src/scheduler/prompt_support.rs:8` — `TONE_AND_STYLE`, `HARD_BLOCKS`, `SOFT_GUIDELINES`, `SKILLS_GUIDANCE`
- `orchestrator/src/scheduler/presets/sisyphus/prompt.rs:1` — imports from `prompt_support` + `prompt_context`

**定性**：这是 **"分层 + 共享块 + preset 自由组装"** 模式，不是 "各自乱拼"。土律要求的"唯一权威"在这里是指应有一个显式的 `PromptSurfaceAuthority` 作为各层的统一入口和组合点。当前状态：共享块做得不错，但统一权威还不够强。

**建议**：将 `SessionPrompt` 模块显式标记为 "prompt surface 唯一构造权威"，各 preset 的 `build_*_dynamic_prompt()` 通过其组合 API 注入扩展内容，而非直接调用 `prompt_support`。

### 问题 #2：SchedulerSkillRef 未声明是 SkillMeta 的投影 🟡

- `orchestrator/src/scheduler/framework.rs:11` — `SchedulerSkillRef { name, description, category }`
- `skill/src/types.rs:41` — `SkillMeta`（更完整的权威类型）

**定性**：这不是"两套独立真相"的语义重复。`SchedulerSkillRef` 是 `SkillMeta` 在调度器上下文中的简化视图。问题在于 **这个投影关系没有被类型系统或文档显式声明**。长期来看，如果 `SkillMeta` 增加了对调度有意义的字段而 `SchedulerSkillRef` 没有跟进，就会出现语义漂移。

**建议**：
- 在 `SchedulerSkillRef` 的文档注释中声明 `// Projection of agendao_skill::SkillMeta for scheduler consumption`
- 或者让 `SchedulerSkillRef` 直接从 `SkillMeta` 构造（`From<SkillMeta>`），确保投影关系在编译期可追踪

### 后续治理点（非当前问题）

Agent 定义分布在 `agendao-agent`（元定义）、`agent_tree`（运行时拓扑）、CLI `agent_cmd`（管理入口）三个位置。这更接近"三种职责的分层"而非"三份真相的分裂"，当前不足以定性为架构问题。可作为后续治理观察点。

## 第五条：金律 (输出之金，贵在成形，忌多法争刃) — 合规度 92%

### 合规项 ✅

- **唯一输出协议**: `agendao-stage-protocol` 定义 `StageSummary`, `StageStatus`, `StageEvent`, `ExecutionNode` — 三端共享
- **统一事件语义**: `telemetry_event_names` 模块定义所有标准事件名
- **统一输出块**: `agendao-output-blocks` 定义 `OutputBlock`, `BlockTone`, `MessagePhase`
- **唯一"完成"定义**: `StageStatus::Done` 是所有执行完成的唯一判断标准

### 评估

三端 (CLI/TUI/Web) 消费同一套 `StageSummary.status` 判断完成状态，同一套 `OutputBlock` 渲染输出内容。展示可变，契约未漂移。无明显问题。

## 第六条：水律 (回流之水，贵在归藏，忌有显无藏) — 合规度 85%

### 合规项 ✅

- **Memory 系统完整**: `MemoryAuthority` → `MemoryValidationEngine` → `MemoryConsolidationEngine` 形成完整的观察→验证→沉淀链路
- **Typed contract 已存在**: 以下结构构成 memory → prompt 回流的正式契约（补充自 biocheming 审核）：
  - `agendao-types/src/memory.rs:188` — `MemoryRetrievalPacket`
  - `session/src/prompt/mod.rs:366` — `memory_prefetch: Option<MemoryRetrievalPacket>`
  - `session/src/prompt/mod.rs:2024` — `render_memory_prefetch_reminder()`
  - `server/src/routes/session/prompt.rs:379` — persisted snapshot / prefetch packet 加载
- **Tool/Memory/Skill Observations**: `ToolMemoryObservation`, `SkillWriteObservation`, `SkillUsageObservation` 形成观测→回流的桥梁
- **Cache Evidence**: 缓存命中/未命中可追溯到具体层

### 注意项（非问题）

Memory → next-turn prompt 的回流契约**已经存在**（`MemoryRetrievalPacket` + `memory_prefetch` + `render_memory_prefetch_reminder`），不是"缺少契约"。当前状态是：**packet/continuity 契约分散在 session/server/scheduler 三个层，缺少一个更高层的统一命名和单入口**。这与 prompt surface 的问题同构——不是没有权威，而是权威的可见性和统一入口可以更强。

## 优先修复建议

### Priority 1：收口 Prompt Surface 统一入口

```text
agendao-session/src/prompt/mod.rs  ← 显式标记为 "唯一 Prompt Surface 构造权威"
  ├── SystemPrompt (system.rs)     ← 产品 header + env
  ├── SessionPrompt (mod.rs)       ← 消息序列化、compaction、ingress
  └── PresetExtension              ← 各 preset 通过组合 API 注入扩展（而非各自调用 prompt_support）
```

关键改动：将 preset 对 `prompt_support` 的直接调用改为通过 `SessionPrompt` 的组合接口注入，增强统一权威的可见性。

### Priority 2：显式化 Skill 语义投影关系

- 为 `SchedulerSkillRef` 添加文档注释声明投影关系
- 可选：实现 `From<SkillMeta> for SchedulerSkillRef`，确保编译期追踪

### Priority 3：提升 Memory 回流契约的统一可见性

- 将 `MemoryRetrievalPacket` → `memory_prefetch` → `render_memory_prefetch_reminder` 这条链路包装在单一的 `MemoryReflow` 命名下
- 不是新建类型（现有 typed contract 已足够），而是通过模块组织结构提升语义可见性

## 正面评价

以下方面做得非常好：

1. **唯一执行内核** (`OrchestrationCore`) 的隔离 — 所有 adapter 经过它，无一例外
2. **ConfigStore** 的 `ArcSwap` 实现 — 土律"唯一配置真相"的教科书级实现
3. **StageProtocol** 的 StageSummary/StageStatus/StageEvent 三层模型 — 金律"唯一输出语法"的正确实践
4. **Ingress 归一化** — 木律"输入唯一收口"的正确实践
5. **Preset 共享构建块** (`prompt_support`, `prompt_context`) — 团队已意识到 prompt 构造的去重需求
6. **Memory 系统的三层处理** (observation → validation → consolidation) — 体现了水律"回流需裁决"的思想
7. **Memory → prompt 回流的 typed contract** (`MemoryRetrievalPacket` 等) — 水生木闭环已有正式类型契约

**总体评分: 88/100**（修订后）— 一套认真践行治理哲学、typed contract 已具雏形的架构。主要改进空间在提升 prompt surface 权威的显式统一入口和 skill 投影关系的显式声明。
